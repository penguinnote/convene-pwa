# Convene — 제품 케이스 스터디

> 다중일 오프라인 행사(워크숍·수련회·캠프)용 설치형 컴패니언 PWA
> 첫 배포: 로뎀나무교회 청년대학부 여름말씀캠프 (80~100명, 3박 4일)
> 역할: **기획·디자인·개발·운영 단독 진행** · 상태: 개발 완료, 2026.07.29~08.01 캠프 배포 예정
> Live: https://rodemcamp.web.app · 코드: [README](README.md) · 디자인 기록: [DESIGN.md](DESIGN.md)

---

## TL;DR

여러 날 진행되는 오프라인 캠프에서 공지·일정·자료가 카톡과 인쇄물에 흩어져 묻히는 문제를, **QR로 설치하고 운영진 공지를 전원 폰에 실시간 푸시하는 PWA**로 풀었다. 앱스토어 심사 없이 배포하고, 캠프장 네트워크가 끊겨도 열리며, 행사의 정체성은 설정 파일 하나로 분리해 재사용 가능하게 만들었다. 성공은 설치율·공지 도달·핵심 기능 완료율로 측정한다(캠프 후 업데이트).

---

## 1. 배경과 문제

캠프는 80~100명이 3박 4일을 함께 보내는데, 정보 전달이 세 갈래로 새고 있었다.

- **공지가 묻힌다** — 단체 카톡방에 공지가 잡담에 밀려 내려가고, 정작 필요한 순간에 못 찾는다.
- **정보가 흩어진다** — 일정은 인쇄물, 방배정은 엑셀 캡처, 말씀 본문은 또 다른 파일. 참가자가 매번 다른 곳을 뒤진다.
- **변경이 반영 안 된다** — 캠프는 집회가 길어지고 식사가 당겨져 순서가 수시로 밀린다. 인쇄물은 이미 손을 떠났다.

**문제 정의:** "오프라인 캠프의 흩어진 정보와 실시간 변경을, 참가자가 한 곳에서, 알림으로, 오프라인에서도 받게 한다."

기존 대안(카톡·밴드·인쇄물)은 각각 실시간성·정보 통합·오프라인 중 하나씩을 놓쳤고, 네이티브 앱은 80~100명에게 설치를 강요하기엔 마찰(심사·용량·업데이트)이 컸다. 그래서 **설치형 PWA**를 택했다.

---

## 2. 목표와 성공 지표 (사전 정의)

배포로 끝내지 않기 위해 캠프 전에 지표를 먼저 정했다. 목표(가설)이며 실제 값은 캠프 후 채운다.

| 갈래 | 지표 | 목표(가설) |
| --- | --- | --- |
| 확보 | 설치율 = 설치 수 / 참석자 수 | ≥ 90% |
| 활성화 | 온보딩(이름·목장) 완료율 | ≥ 90% |
| 활성화 | 조 편성 완료율(레크레이션 참여자 중) | ≥ 80% |
| 참여 | 일자별 재방문(4일 리텐션) | 3일 이상 방문 ≥ 60% |
| 알림 | 푸시 권한 허용률 | ≥ 70% |
| 알림 | 공지 푸시 → 열람 전환 | 정성적으로 "빠뜨리는 사람 없음" |

---

## 3. 사용자·이해관계자

- **참가자(80~100명)** — "지금 뭐 하지? 내 방은? 다음 말씀은?"을 마찰 없이. 온보딩은 이름·목장만, 그 외엔 열람 중심.
- **운영진(관리자)** — 공지를 즉시 전원에게, 캠프 중 방배정·조 편성·일정 변동을 **노트북 없이 폰에서** 통제.
- **리더십(목회자)** — 캠프가 매끄럽게 굴러가고, 참여가 실제로 일어나는지.

세 층의 니즈가 달라 화면과 권한을 분리했다(참가자 열람 / 관리자 콘솔 / 측정 지표).

---

## 4. 해결책 개요

한 앱에 **실시간 공지 푸시**, **라이브 "현재 순서"**(지금·다음 순서 자동 표시 + 관련 자료 바로가기), **레크레이션 조 편성 게임**, **참여자 디렉토리**, **콘텐츠 열람**(일정·방배정·말씀 3역본·식단·플레이리스트), **관리자 콘솔**을 담았다. QR로 홈 화면에 설치되고 오프라인에서 열리며, 넓은 화면(태블릿·노트북)엔 별도 데스크톱 레이아웃을 제공한다.

---

## 5. 핵심 제품 결정과 트레이드오프

포트폴리오의 핵심. 각 결정은 "문제 → 선택지 → 결정 → 근거"다.

**① 라이브 "현재 순서" — 자동 vs 수동 (3회 반복)**
기기 시계 자동 계산으로 시작했으나 순서가 10~40분씩 밀려 "지금"을 자주 틀렸다. 관리자 수동 포인터로 바꿨더니 매 순서 넘기는 마찰·누락이 컸다. 최종은 **시각 자동 계산 + 관리자 변동 메모** 절충 — 큰 흐름은 자동으로 맞추고, 밀림·당김 예외만 사람이 한 줄("점심 30분 지연")로 덧댄다. 자동의 무마찰과 수동의 정확성을 나눠 가졌다.

**② 데이터 위치 — 정적 vs 실시간, 그리고 무배포 운영**
불변·읽기비용 데이터(일정·말씀)는 정적 파일에, 가변·실시간 데이터(공지·프로필·라이브 메모)는 Firestore에 두어 비용을 최소화했다. 여기에 **방배정은 예외로 Firestore(단일 문서)로 이전** — iOS PWA는 코드 배포 반영이 늦어 캠프 중 재설치를 요구하는데, 방배정은 자주 바뀌므로 "관리자가 폰에서 저장 → 전원 즉시 반영, 무배포"가 미미한 읽기비용보다 훨씬 값졌다. 원칙을 세우되 근거가 뒤집힐 땐 예외를 명시했다.

**③ 조 편성 — 재미 vs 인원 균형**
취향 질문 4개로 팀을 나누는 게임. 답변대로 결정하면 인원이 쏠릴 수 있고 카운터로 균형을 맞추면 서버 로직이 필요하다. **"같은 취향 = 같은 팀"의 재미를 택해** 답변↔구역을 전단사로 매핑, 서버 없이 결정하고 결과에 선택지까지 되복원해 보여 준다. 오락 도구의 목적(아이스브레이킹)에 맞춰 정합성보다 경험을 우선했다.

**④ 로그인 — 익명 유지 vs 구글 전환**
참여자 커뮤니티·조 편성 자동 채움을 위해 로그인 활용을 넓히면서도, 80~100명 전원에게 구글 로그인을 강요하는 마찰을 피했다. 기능이 필요한 건 "검증된 신원"이 아니라 "일관된 이름·목장"이라 판단, **익명 로그인 유지**로 마찰 0을 지키고 중복은 관리자 도구로 정리했다.

**⑤ 스코프 컷 — 뺀 것의 근거**
댓글은 운영 부담·부정적 상호작용 위험으로 제거했고, 참여자 "팔로우/팔로잉"은 관계 그래프 없이 인원수 연출만으로 "모두가 동역자" 메시지를 최소 복잡도로 냈다. 식단표는 콘텐츠 미확정이라 스캐폴드만 두고 보류했다.

**⑥ 보안 — 익명 노출 발견과 다층 방어**
전원 익명 로그인 구조 탓에 "로그인 여부"만으론 관리자를 가릴 수 없어 익명 사용자에게 관리자 화면이 노출되는 결함이 있었다. UI 게이트를 이메일 보유 기준으로 바꾸고 Firestore 규칙에서 쓰기를 이메일 관리자로 제한해 **화면과 데이터 양쪽**을 막았다.

## 6. 릴리스와 운영

- **배포**: QR 스캔 → 홈 화면 설치. 앱스토어 심사·업데이트 없이 웹 배포로 갱신.
- **iOS PWA 제약 대응**: 16.4+ 설치 실행 상태에서만 푸시가 되는 제약에 온보딩을 맞추고, 배지·중복 푸시·서비스워커 갱신을 플랫폼별로 처리.
- **캠프 중 무배포 운영**: 방배정·조 편성·공지·라이브 메모를 관리자가 폰에서 실시간 편집.
- **권한 위임**: 임원 계정을 추가해 여러 운영진이 관리자 콘솔을 공유.

## 7. 측정 계획

자체 이벤트 로그(`events` 컬렉션)에 설치·열람·온보딩 완료·조 편성·공지 열람·라이브 링크 클릭·푸시 권한을 플랫폼·세션·일자와 함께 남긴다. 캠프 후 이 로그로 2장의 지표를 산출하고, 필요 시 Google Analytics 4를 병행해 퍼널·리텐션 대시보드를 얹는다. (이벤트는 쓰기 전용·관리자만 조회.)

## 9. 배운 점 (개발 단계)

- **트레이드오프를 문서로 남기니 결정이 빨라졌다.** 라이브 카드처럼 되돌아온 결정도 근거가 있으면 재논의 비용이 낮다.
- **제약이 설계를 이끈다.** iOS PWA 푸시·배포 제약이 온보딩과 "무배포 실시간 편집"이라는 운영 설계를 낳았다.
- **원칙과 예외를 함께 명시한다.** "정적 우선"을 세우되 방배정처럼 근거가 뒤집힐 땐 예외를 기록해 팀(과 미래의 나)이 헷갈리지 않게 했다.
